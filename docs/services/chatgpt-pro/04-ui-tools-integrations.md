## 개발자 관점 핵심
이 챕터의 목적은 “툴이 있냐 없냐”가 아니라, **언제 어떤 흐름으로 붙일지**를 정하는 것입니다.

## 통합 패턴

| 패턴 | 입력 | 출력 | 추천 상황 |
|---|---|---|---|
| Web 단독 | 요구사항/코드 조각 | 설계안/초안 | 빠른 사고 정리 |
| Web + 로컬 IDE | diff/로그/테스트 결과 | 수정 제안 + 테스트 항목 | PR 준비 |
| API + CI | 이슈/커밋/테스트 리포트 | 자동 코멘트/체크리스트 | 반복 업무 자동화 |

## IDE/PR 연계 프롬프트
```text
다음 PR diff를 리뷰해줘.
출력 형식:
- 위험도(높음/중간/낮음)
- 버그 가능성 3개
- 성능 포인트 2개
- 테스트 보강안 3개
- 바로 복붙 가능한 리뷰 코멘트
```

## 로그 분석 연계 프롬프트
```text
아래 에러 로그를 기반으로
1) 가능한 원인 가설
2) 우선 검증 순서
3) 임시 완화책
4) 영구 수정안
으로 표로 정리해줘.
```

## 주의점
- 외부 연동 기능은 계정/권한/정책에 따라 차이 큼 (`확인 필요`)
- 자동화 적용 전에 반드시 샌드박스/스테이징에서 검증

## 체크리스트
- [ ] 연동 목적(리뷰/분석/자동화)을 먼저 정했는가
- [ ] 입력 데이터의 민감정보를 제거했는가
- [ ] 자동화 적용 전 수동 검증 단계가 있는가
