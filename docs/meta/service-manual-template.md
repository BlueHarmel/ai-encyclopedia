# Service Manual Template

Use this template when adding or refactoring a service manual. The goal is a practical manual backed by official sources, not a generic product introduction.

## Required Questions

- What role does this service play in current workflows?
- Where are official facts confirmed?
- Where do release notes or changelogs enter?
- Where are plan, pricing, account, and billing facts confirmed?
- How are app behavior, API behavior, model behavior, and tool behavior separated?
- How does this connect to our PC, OpenClaw, Brain Vault, or publishing workflow?
- Which facts still lack official evidence?

## Required Pages

- Overview
- Plans or Pricing
- Models
- Tools
- App, API, or surface boundary
- Workflows
- Release Notes or Changelog
- Coverage

## Page Rules

- Put facts on the page where a reader would naturally look for them.
- Keep official source URLs, last checked dates, and coverage status visible.
- Treat X posts, news, blogs, and community posts as signals until an official source confirms a product fact.
- Mark gaps as `needs-recheck` or `needs-extraction` instead of hiding them.
- Store all public Markdown as UTF-8.
