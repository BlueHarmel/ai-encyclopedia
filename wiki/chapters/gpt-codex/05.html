5. 기능 카탈로그(실사용 매뉴얼)

빠른 선택표: 작업 유형별 권장 기능

기능적합한 상황입력 핵심기대 출력실패 패턴
범위 제한 패치 생성 운영 코드의 작은 hotfix 또는 리뷰 가능한 작은 PR 허용/금지 파일 + 성공조건 최소 diff + 변경 이유 + 테스트 제안 범위 미지정으로 주변 모듈까지 과변경
실패 재현 테스트 생성 재발성 장애 원인을 고정해야 할 때 오류 로그 + 재현 단계 + 기대 동작 선실패 테스트 + 수정 패치 + 재검증 절차 정상 케이스만 작성하고 회귀 테스트 누락
공식 변경로그 triage 월간 업데이트를 팀 액션으로 전환할 때 공식 changelog/release 링크 + 현재 파이프라인 즉시/파일럿/보류 분류표 비공식 요약 글을 근거로 운영 정책 변경

기능 1) 범위 제한 패치 생성

실행 목적: 큰 리팩터링이 아니라 "리스크를 통제한 최소 변경"을 만드는 것.

요청 템플릿
목표: 로그인 실패 메시지 표준화
허용 변경: src/auth/error-map.ts, src/auth/error-map.test.ts
금지 변경: migrations/, infra/, package-lock.json
성공 조건: npm test -- error-map 통과
출력 형식: unified diff + 변경 이유 3줄

실제 사용 예시 1: 릴리즈 전날 오류 메시지 포맷만 급히 통일해야 할 때, 위 템플릿으로 변경 범위를 잠그고 리뷰 시간을 줄일 수 있다.

  • 허용 파일 외 변경 0건 확인
  • 테스트 명령 재실행 결과 첨부
  • 요구사항-변경점 1:1 매핑 검토

기능 2) 실패 재현 테스트 생성

실행 목적: "고친 것 같은데 재발" 문제를 테스트로 고정.

요청 템플릿
상황: 결제 취소 API가 특정 금액에서 500
입력: 오류 로그, 재현 단계, 함수 시그니처
요구: 정상/예외/경계 테스트 3종
출력: 실패 재현 테스트 -> 수정안 -> 재검증 절차

실제 사용 예시 2: 운영 장애 회고에서 원인 불명으로 끝났던 이슈를 선실패 테스트로 고정해, 다음 배포에서 같은 결함을 자동으로 차단.

  • 수정 전 테스트 실패 여부
  • 수정 후 동일 테스트 통과 여부
  • 경계값(0/최대/음수) 포함 여부

기능 3) 공식 변경로그 triage(즉시/파일럿/보류)

실행 목적: 업데이트 정보를 팀 실행 단위로 변환.

입력출력검토 포인트
공식 changelog/release 링크 + 현재 운영 파이프라인 즉시 적용/파일럿/보류 + 재검토 날짜 모든 판단에 공식 링크를 1:1로 매핑

실제 사용 예시 3: 월초 점검 회의에서 변경 12건을 triage해 즉시 2건, 파일럿 4건, 보류 6건으로 분리하고 담당자를 지정.
