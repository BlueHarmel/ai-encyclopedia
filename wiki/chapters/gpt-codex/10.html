10. 제한사항·주의점(검증 가능한 범위)

주의점 1) 생성 코드의 신뢰 한계

  • 생성 코드는 초안으로 간주하고 테스트/리뷰 후 반영한다.
  • 한 번에 큰 범위 변경을 적용하지 않고 기능 단위로 분리한다.
예시 블록 1 — 작은 PR 원칙

한 PR에 1개 기능만 담으면 장애 발생 시 원인 추적과 롤백이 훨씬 단순해진다.

예시 블록 2 — 테스트 없는 제안 보류

코드 제안만 있고 재현 테스트가 없다면 즉시 머지하지 않고 "검증 대기" 라벨로 분리한다.

주의점 2) 보안/운영 명령 처리

  • 파괴적 명령은 자동 실행하지 않고 제안 중심으로 처리한다.
  • 배포 전 백업/롤백 경로를 확보한다.
  • 민감정보(토큰/비밀번호)는 입력 전에 마스킹한다.
예시 블록 3 — dry-run 기본값

운영 스크립트는 dry-run을 기본으로 두고, 실제 실행은 승인 후 단계적으로 전환한다.

예시 블록 4 — 비밀정보 필터링

로그/프롬프트에 API 키가 포함되지 않도록 사전 필터를 적용하면, 협업 채널 유출 위험을 줄일 수 있다.

주의점 3) 정책/플랜 변동성

  • 과금/한도/기능 범위는 시점별로 바뀔 수 있다.
  • 공식 Source + 검증됨 링크 기준으로 주기 점검한다.
  • HTTP 403 등으로 검증이 막히는 항목은 단정 문장에서 제외한다.
예시 블록 5 — 검증 불가 분리

문서 접근이 제한된 링크는 "확인 불가/직접 확인 필요"로 관리하고, 운영 정책 결정 근거에서는 제외한다.

예시 블록 6 — 월간 재검증 루틴

월 1회 changelog/release 링크 상태를 재점검해, 만료되거나 변경된 근거를 최신 상태로 유지한다.

확인 불가/직접 확인 필요: OAuth/API 과금 세부, 조직별 기능 제한, 시간 단위 한도 정책.
