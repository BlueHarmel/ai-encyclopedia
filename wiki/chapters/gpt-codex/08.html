8. 중급 예제(실무 활용)

예제 1) 운영 장애 대응: "재현 테스트 먼저" 패턴

요소내용
상황간헐적 500 에러가 재현되지 않아 같은 장애가 반복된다.
입력오류 로그 샘플, 재현 단계, 허용 변경 파일, 성공 조건(테스트 통과)
출력선실패 테스트 + 최소 수정 패치 + 재검증 명령
검토포인트수정 전 실패/수정 후 통과가 명확한지, 허용 파일 외 변경이 없는지
실패패턴테스트 없이 코드를 먼저 바꿔 원인 추적이 불가능해짐
실행 메모(실제 운영)
1) 에러 로그를 그대로 첨부하지 말고 민감값 마스킹
2) "재현 테스트 우선"을 요청문에 명시
3) 패치 반영 전 CI 통과 스냅샷 첨부

예제 2) 월간 변경로그 triage를 팀 백로그로 변환

요소내용
상황공식 변경로그를 읽지만 실제 반영 우선순위가 매번 달라진다.
입력월간 changelog/release 링크, 현재 파이프라인, 배포 정책
출력즉시/파일럿/보류 분류표 + 담당자 + 재검토 날짜
검토포인트분류 판단에 공식 링크 근거가 1:1로 붙어 있는지 확인
실패패턴제목만 보고 즉시 반영해 운영 환경 비호환 이슈 발생
운영 팁
- 즉시: 리스크 낮고 테스트 가능한 항목
- 파일럿: 영향 범위가 큰 항목(저장소 1개 선적용)
- 보류: 근거/효과가 불명확하거나 정책 확인이 필요한 항목

예제 3) PR 본문 자동화 + 리뷰 질문 자동 생성

요소내용
상황PR 설명 품질 편차 때문에 리뷰 시간이 늘어난다.
입력변경 파일 목록, 테스트 결과, 위험 구간(보안/성능/호환성)
출력PR 본문(배경/변경점/리스크/검증) + 리뷰 질문 3~5개
검토포인트리뷰어가 "어디를 집중해서 봐야 하는지" 바로 알 수 있는지 확인
실패패턴자동 생성 본문을 무검토로 merge해 누락 정보 발생
요청 예시
- 목표: 리뷰어가 5분 안에 핵심 리스크 파악
- 형식: [변경 배경][주요 diff][테스트 결과][롤백 경로][리뷰 질문]
