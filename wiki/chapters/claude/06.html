6. 인터페이스별 사용 흐름(Web/앱/API/CLI 해당 시)

Web/앱 흐름

  • 1단계: 작업 목적(요약/검토/공지)과 독자(임원/실무/고객)를 먼저 고정
  • 2단계: 출력 형식을 사전 지정(예: 결론 3개 + 액션 5개 + 확인 필요 2개)
  • 3단계: 단정 문장은 근거 링크와 함께 확인, 근거 부족은 별도 분리
예시 블록 1 — 회의록 변환

같은 회의록을 "임원용 8줄"과 "실무 액션리스트"로 분리해 요청하면, 한 번의 입력으로 보고/실행 문서를 동시에 만들 수 있다.

예시 블록 2 — 고객 공지 초안

공지문 생성 시 "확정 정보/검토 중 정보"를 분리해 작성하면, 추후 정정 공지 리스크를 줄일 수 있다.

API 흐름

  • 1단계: 시스템 지시에서 역할·톤·금지사항(추정 단정 금지) 고정
  • 2단계: 응답 스키마를 요약/근거/확인필요로 고정
  • 3단계: 확인필요 섹션만 재질의하는 2-pass 루프로 검증
예시 블록 3 — 문서 파이프라인

초안 생성 → 근거 누락 점검 → 최종 문체 변환의 3단계 파이프라인을 쓰면 품질 편차를 줄일 수 있다.

예시 블록 4 — FAQ 자동화

자주 묻는 질문 10개를 먼저 만들고, 정책·플랜 관련 답변은 "직접 확인 필요" 태그를 자동 부여해 배포 전 검토 큐로 보낸다.

CLI 관련 주의

  • sources.json 기준으로 범용 Claude CLI의 기능/정책을 단정할 공식 근거는 제한적이다.
  • CLI 세부 동작은 "확인 불가/직접 확인 필요"로 유지하고 공식 문서 확인 후 반영한다.

Source: Claude Docs Overview, Claude API Docs Overview, Anthropic News(검증됨 링크는 13번 문서 참조)

확인 불가/직접 확인 필요: 계정/플랜별 세부 기능 제공 범위, 사용량 한도, 조직 정책 차이.
