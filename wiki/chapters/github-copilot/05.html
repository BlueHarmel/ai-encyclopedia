GitHub Copilot

5. 핵심 기능별 설명(입력/출력/옵션/실패 패턴)

기능 A) 코드 제안(자동 완성/블록 제안)

  • 입력: 함수 목적, 입출력 조건, 금지사항(예: 특정 라이브러리 미사용), 기존 코드 컨텍스트
  • 출력: 라인 또는 블록 단위 후보 코드
  • 옵션: 제안 반복 생성, 일부만 채택, 수동 수정 후 재요청
  • 실패 패턴: 요구가 추상적이면 의도와 다른 구현이 섞일 수 있음
구체 예시 1 — 입력 제약을 먼저 적는 방식

VAT 계산 함수, 반올림은 소수점 둘째 자리, 음수 입력은 예외처럼 제약을 먼저 주면 재작성 횟수가 줄어든다. 목적만 적었을 때보다 검토 diff가 짧아지는 편이다.

구체 예시 2 — 변경 금지 구역 명시

legacy/ 폴더는 읽기 전용으로 두고 service/price.ts만 수정 허용한다고 명시하면, 제안 범위를 좁혀 회귀 위험을 줄일 수 있다.

기능 B) 테스트/리뷰 보조(코드 채택 전 검증 강화)

  • 입력: 대상 함수, 정상/예외/경계 케이스, 실패 로그(있다면)
  • 출력: 테스트 초안 + 리뷰 체크포인트(누락 조건, 리스크 항목)
  • 옵션: 경계 케이스 우선, 실패 재현 테스트 선행
  • 실패 패턴: 정상 케이스 위주로만 생성되어 운영 이슈 재현이 빠질 수 있음
구체 예시 1 — 결제 검증 로직

정상 케이스 외에 0원, 음수, 상한 초과를 먼저 생성하도록 지시하면, 배포 전 실패 시나리오를 빠르게 걸러낼 수 있다.

구체 예시 2 — PR 본문 자동 초안

"배경/변경/검증/리스크" 4칸 템플릿으로 PR 설명을 먼저 만들고 사람이 마지막으로 압축 편집하면, 리뷰 질문 왕복을 줄이는 데 유리하다.

기능 C) 공식 변경사항 기반 운영 반영(Changelog 연계)

  • 입력: Copilot changelog 항목, 현재 조직 정책, IDE/플랜 상태
  • 출력: "즉시 시도 / 파일럿 / 보류" 분류표
  • 옵션: 공개 프리뷰 항목 분리, 팀 표준 템플릿으로 기록
  • 실패 패턴: changelog 제목만 보고 전사 적용을 결정
구체 예시 1 — changelog triage 30분 루프

주 1회로 changelog를 확인하고 각 항목을 즉시 시도/파일럿/보류로 나눈다. 회의록에는 "근거 링크 + 적용 범위 + 재검토 날짜"를 함께 남긴다.

구체 예시 2 — 공개 프리뷰 항목의 단계 적용

공개 프리뷰 표기 항목은 전사 배포 대신 샌드박스 저장소에서 먼저 검증하고, 실패 패턴이 없을 때만 팀 저장소로 확대한다.

기능 D) Quickstart 기반 온보딩 표준화

  • 입력: Quickstart 절차, 팀의 개발 환경 목록(IDE/OS), 첫 주 과제
  • 출력: 신규 인원용 "첫날 설정 체크리스트" + "첫 PR 체크리스트"
  • 옵션: 역할별(백엔드/프론트/데이터) 체크리스트 분기
  • 실패 패턴: 개인별 임의 설정으로 품질 기준이 팀 내에서 흔들림
구체 예시 1 — 첫날 설정 체크리스트

Quickstart 순서대로 설치 확인 → 제안 동작 확인 → 테스트 1건 생성까지 완료하게 하면, "제안이 안 뜬다" 같은 초반 장애를 줄일 수 있다.

구체 예시 2 — 첫 PR 체크리스트

첫 PR에서 자동 제안 채택 이유 1줄 + 수동 수정 이유 1줄을 남기도록 하면, 팀이 Copilot 사용 패턴을 빠르게 교정할 수 있다.
