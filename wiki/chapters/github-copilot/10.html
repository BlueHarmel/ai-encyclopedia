GitHub Copilot

10. 제한사항·주의점(검증 가능한 범위)

주의점 1) 제안 코드의 검토 필요성

  • Copilot 제안은 초안이며 정답이 아니다.
  • 사람 리뷰와 테스트 없이 반영하면 회귀 위험이 커진다.
예시 블록 1 — 리팩터링 반영 규칙

제안된 리팩터링은 "기존 테스트 통과 + 신규 경계 테스트 1개"를 만족할 때만 병합하는 규칙을 두면 안전하다.

예시 블록 2 — 작은 단위 머지

대규모 제안은 기능 단위 PR로 나눠 병합하면 장애 시 원인 추적과 롤백이 쉬워진다.

주의점 2) 보안/라이선스 검증

  • 민감정보(토큰/계정키)가 코드/주석에 노출되지 않도록 점검한다.
  • 새 라이브러리 제안은 라이선스/취약점 검토 후 반영한다.
예시 블록 3 — 시크릿 필터링

PR 전 자동 스캔으로 시크릿 문자열을 확인하면, Copilot 제안 포함 코드의 유출 위험을 줄일 수 있다.

예시 블록 4 — 의존성 승인 절차

외부 패키지 추가 제안은 보안팀/리뷰어 승인 단계를 거쳐야 운영 환경 반영이 가능하도록 제한한다.

주의점 3) 기능 가용성 변동

  • 기능/지원 환경은 GitHub Changelog 시점에 따라 달라질 수 있다.
  • 공식 Docs/Changelog 기준으로 월간 재검토 루프를 유지한다.
예시 블록 5 — Public preview 분리

공개 프리뷰 기능은 전사 즉시 적용 대신 파일럿 저장소에서만 먼저 검증한다.

예시 블록 6 — 정책 불일치 처리

조직 정책으로 비활성화된 기능은 문서에 "확인 불가/직접 확인 필요"로 표시해 오해를 방지한다.

확인 불가/직접 확인 필요: 플랜별 세부 기능, 조직 정책별 제한 범위, 모델 선택 옵션.
