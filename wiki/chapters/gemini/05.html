Gemini

5. 핵심 기능별 설명(입력/출력/옵션/실패 패턴)

기능 A) 문서형 질의응답/요약

  • 입력: 질문 목적, 원문 범위, 출력 형식(불릿/표/보고서), 독자 수준
  • 출력: 핵심 요약 + 근거 기준 + 확인 필요 항목 분리
  • 옵션: 길이 제한, 항목 수 제한, "결정사항/미결사항" 구분
  • 실패 패턴: 목표를 안 주면 장황한 일반론으로 흐르기 쉬움
구체 예시 1 — 운영 보고 1페이지 변환

입력을 회의록 그대로 넣기보다 핵심 3개 / 영향 2개 / 다음 액션 3개 형식으로 고정하면, 의사결정 문서로 재활용 가능한 결과가 나온다.

구체 예시 2 — 공지문 2버전 동시 생성

같은 내용을 "경영진 8줄"과 "실무 체크리스트"로 분리 출력하면, 전달 대상별 문장 톤 차이를 한 번에 처리할 수 있다.

기능 B) 멀티모달 입력 해석(텍스트+파일/이미지)

  • 입력: 텍스트 지시문 + 파일/이미지(지원 범위는 공식 Docs/업데이트 기준)
  • 출력: 관찰 사실 / 해석 / 추정 분리 결과
  • 옵션: JSON 스키마 고정, 섹션 강제, 금지 추론 규칙
  • 실패 패턴: 관찰과 추정이 섞여 단정형 문장이 생성됨
구체 예시 1 — 이미지 검토 리포트

"보이는 것만 먼저" 규칙을 걸어두면 추측 문장을 줄일 수 있다. 이후 별도 섹션에서만 해석을 허용하면 검토가 쉬워진다.

구체 예시 2 — 텍스트+첨부 결합 검토

정책 문서와 스크린샷을 함께 넣을 때, 문서 근거와 화면 관찰을 분리하면 책임 있는 검증 경로를 유지할 수 있다.

기능 C) 공식 업데이트 추적 후 문서 재정리

  • 입력: 공식 Docs/Quickstart/Google Blog/Gemini Updates 링크, 기존 문서
  • 출력: 변경점 요약 + 영향 범위 + 적용 우선순위
  • 옵션: "즉시 적용/파일럿/보류" 라벨 분류
  • 실패 패턴: 비공식 글을 공식 업데이트처럼 혼합
구체 예시 1 — Gemini updates 반영

gemini.google.com/updates에서 확인된 항목만 단정 문장으로 반영하고, 중복/분리 불가 항목은 "확인 불가"로 분리한다.

구체 예시 2 — API 변경 공지 반영

Google 공식 블로그의 API 변경 공지가 있으면, 내부 가이드에 "영향 받는 입력 크기/형식" 체크리스트를 먼저 추가한 뒤 코드 반영을 진행한다.

Source: Gemini API Docs, Gemini API Quickstart, Google 공식 Gemini 블로그, Gemini Updates(검증됨)

확인 불가/직접 확인 필요: 플랜별 기능 범위, 지역별 제한, API 과금 세부, 조직 정책별 활성화 차이.
